# Суверенная Mesh-VPN — Документация и дорожная карта

> Рабочее название: **ArmMesh** (плейсхолдер, переименуй на своё).
> Стек: **Rust**. Цель: независимый законченный продукт + максимальный рост в сетях и прикладной криптографии.

---

## 1. Что мы строим (и чего не строим)

**Что это.** Самохостящийся зашифрованный overlay-network. Лёгкий демон ставится на твои устройства (телефон, ноут, сервер), и они оказываются в одной приватной сети, будто стоят на одном LAN, где бы ни находились физически. Трафик между ними end-to-end зашифрован и идёт поверх обычного интернета.

**Чего это НЕ.**
- Не замена интернета и не «свой отдельный интернет» — это слой *поверх* существующего.
- Не новый мессенджер. Это сетевая инфраструктура.
- Не переписанный WireGuard ради переписывания (подробно — раздел 4).

**Зачем (ценность).**
- Полная суверенность: ключи у тебя, координатор разворачиваешь сам, нулевое доверие третьим серверам.
- Совпадает с твоими ценностями и долгой целью — уважаемый открытый security-продукт это самостоятельный козырь.
- Качает ровно твой edge: прикладная крипта + сети + системное программирование.

---

## 2. Модель угроз (читать первым)

Это центральная часть. Продукт без явной модели угроз — это не security-продукт, а игрушка. Формула: **от кого именно и что конкретно мы защищаем.**

### От кого защищаем
- Интернет-провайдер пользователя.
- Любой промежуточный узел на пути (Wi-Fi точка, транзитные сети).
- Пассивный наблюдатель, который пишет трафик.

### Что закрываем
- **Содержимое** трафика между твоими устройствами — полностью.
- **Метаданные назначения** — провайдер видит только зашифрованный поток к одному endpoint, но не знает, какие сайты/сервисы за ним (в режиме exit-узла).
- Унифицируем шифрование: всё выглядит одинаково, нет «голых» протоколов.

### Что НЕ закрываем (честные границы)
- **Оператор exit-узла видит трафик на выходе.** В самохостинге это ты сам — нормально. Но это надо понимать: VPN *переносит доверие*, а не уничтожает его.
- **Сайт назначения** всё равно видит запрос (с IP exit-узла).
- **Traffic analysis** — объём и тайминги пакетов могут что-то выдать даже при шифровании.
- Скомпрометированное устройство-участник. Endpoint security — вне периметра.

### Допущения
- Криптопримитивы (Noise, ChaCha20-Poly1305, Curve25519) считаем стойкими.
- Пользователь способен безопасно хранить приватный ключ устройства.

> Важно: помни разницу между device-to-device mesh (защищает трафик *между твоими устройствами*) и exit-режимом (прячет *весь сёрфинг* от провайдера). Это разные функции продукта.

## 3. Архитектура

```
   Приложения / ОС
        │
   ┌────▼─────┐   виртуальная сетевая карта
   │   TUN    │   ОС думает, что это обычный интерфейс
   └────┬─────┘
        │  IP-пакеты
   ┌────▼──────────────┐
   │  ArmMesh daemon    │
   │  ┌──────────────┐  │
   │  │ Фрейминг /    │  │  ← ТВОЙ протокол (формат, мультиплекс)
   │  │ транспорт     │  │
   │  ├──────────────┤  │
   │  │ Noise + AEAD  │  │  ← проверенная крипта, НЕ изобретаешь
   │  └──────────────┘  │
   └────┬──────────────┘
        │  зашифрованные датаграммы
   ┌────▼─────┐
   │   UDP    │
   └────┬─────┘
        │
   Обычный интернет ──► другой узел (обратный порядок)
```

### Два плоскости (запомни это деление)

**Data plane** — то, что гоняет зашифрованные пакеты между узлами. Здесь живут: UDP, твой фрейминг, Noise-хендшейк, AEAD, TUN.

**Control plane** — то, что делает сеть *управляемой*: кто в сети, обмен публичными ключами, обнаружение пиров, координация, NAT traversal, ACL. **Здесь вся реальная ценность продукта** (см. Tailscale: они не переписывали WireGuard, они построили именно control plane).

### Компоненты
1. **TUN-интерфейс** — точка стыка с ОС. Заворачивает выбранный трафик в демон.
2. **Транспорт/фрейминг** — формат пакетов, версии, типы сообщений, мультиплексирование, keepalive. Это твой протокол.
3. **Криптослой** — Noise (паттерн IK, как в WireGuard) для рукопожатия и обмена ключами; ChaCha20-Poly1305 для данных.
4. **Пир-менеджер** — таблица пиров, их публичные ключи, endpoints, состояние сессий, ротация ключей.
5. **Координатор (control plane)** — самохостящийся сервис: регистрация узлов, раздача актуального списка пиров и ключей. Аналог Tailscale coordination server / Headscale.
6. **NAT traversal** — UDP hole punching + STUN-подобное определение внешнего адреса + relay-фоллбэк, когда прямое соединение невозможно.

## 4. Стратегия: два трека (не смешивай)

Самая частая ловушка сильных инженеров — закопаться в переписывание уже решённого и не дойти до того, что реально отличает продукт. Поэтому раздели явно:

### Трек A — Learn (свой транспорт с нуля)
Пишешь свой WireGuard-подобный транспорт (UDP + Noise + фрейминг + TUN) сам.
- **Цель:** понять сети и крипту до самого дна; артефакт-доказательство компетенции.
- **Не для прода без аудита.** Это учебно-исследовательский трек.
- Здесь же реализуешь Noise по спецификации (это твой «крипто-автор» уровень).

### Трек B — Ship (продукт на WireGuard)
Берёшь готовый, проверенный data plane (`boringtun` — userspace WireGuard на Rust от Cloudflare, или kernel WireGuard) и строишь продукт *вокруг* него: control plane, обнаружение пиров, NAT traversal, UX, самохостинг.
- **Цель:** довести независимый продукт до релиза, не утонув в рисках своей крипты.
- Здесь живёт вся пользовательская ценность и дифференциация.

### Как сочетать
Сначала Трек A — чтобы *понимать кишки*. Потом Трек B — чтобы *выпустить*. Свой транспорт переносишь в прод только если позже пройдёшь независимый аудит. Это та же логика, что я давал по ArmSecure: код — меньшая половина, доверие — большая.

## 5. Технологический стек

| Слой | Крейт / технология |
|---|---|
| Async runtime | `tokio` |
| UDP | `tokio::net::UdpSocket` |
| Noise framework | `snow` (чистый Rust Noise) |
| Data plane для Трека B | `boringtun` (userspace WireGuard) |
| TUN-устройство | `tun` / платформенные обёртки |
| AEAD | `chacha20poly1305` (RustCrypto) |
| Ключи | `x25519-dalek`, `ed25519-dalek` |
| Сериализация control plane | `serde` + `bincode` / `prost` (protobuf) |
| CLI | `clap` |
| Логи/трейсинг | `tracing` |

## 6. Дорожная карта по фазам

Каждая фаза имеет **жёсткий критерий готовности** — это против твоего главного риска (не доводить до конца). Не переходи дальше, пока критерий не выполнен.

### Фаза 0 — Скелет (выходные)
- [x] Демон на Rust поднимает TUN-интерфейс.
- [x] Берёт IP-пакеты из TUN, шифрует **статическим** ключом (ChaCha20-Poly1305), шлёт по UDP на второй узел.
- [x] Вторая сторона расшифровывает и пишет обратно в свой TUN.
- **Готово, когда:** `ping` между двумя машинами реально проходит через твой зашифрованный тоннель. Это твоя первая победа — она потащит дальше.

### Фаза 1 — Настоящее рукопожатие
- [ ] Заменить статический ключ на Noise-хендшейк (паттерн IK) через `snow`.
- [ ] Обмен публичными ключами устройств, установление сессии.
- [ ] Ротация сессионных ключей по времени/объёму.
- **Готово, когда:** два узла поднимают сессию через Noise, перехваченный трафик нечитаем, рестарт корректно пересогласует ключи.

### Фаза 2 — Несколько пиров
- [ ] Таблица пиров (ключ → endpoint → состояние сессии).
- [ ] Маршрутизация: по dst-IP выбрать нужного пира.
- [ ] Keepalive и обнаружение «мёртвых» сессий.
- **Готово, когда:** 3+ устройства в одной mesh-сети, любой пингует любого.

### Фаза 3 — Control plane (минимальный координатор)
- [ ] Самохостящийся сервис: узлы регистрируются, получают актуальный список пиров и их публичные ключи.
- [ ] Аутентификация узла при регистрации.
- [ ] Конфиг-файл устройства (свой ключ, адрес координатора).
- **Готово, когда:** новое устройство добавляется в сеть без ручной правки конфигов на остальных.

### Фаза 4 — NAT traversal
- [ ] Определение внешнего адреса (STUN-подобно).
- [ ] UDP hole punching между пирами за NAT.
- [ ] Relay-фоллбэк, когда прямое соединение невозможно (аналог DERP).
- **Готово, когда:** два устройства за разными домашними роутерами соединяются напрямую без проброса портов.

### Фаза 5 — Режимы и продуктовость
- [ ] Exit-узел: маршрутизация всего трафика через выбранный узел (приватность сёрфинга от провайдера).
- [ ] Subnet routing: доступ к домашней подсети удалённо.
- [ ] Нормальный CLI/демон UX, автозапуск, статус-команды.
- **Готово, когда:** из-за пределов домашней сети заходишь в домашние сервисы и выходишь в интернет «как из дома».

### Фаза 6 — Релиз и доверие
- [ ] Документация для пользователя, инструкция по самохостингу координатора.
- [ ] Тесты (юнит + интеграционные на поднятие сети), CI.
- [ ] Открытый репозиторий, README, лицензия.
- [ ] План на независимый аудит, если планируешь свой транспорт в прод.
- **Готово, когда:** незнакомый человек разворачивает сеть по твоей инструкции без твоей помощи.

## 7. Чего НЕ делать (якоря здравого смысла)

- **Не пиши свою крипту.** Сам дизайн крипто-хендшейка не изобретаешь — берёшь Noise. Самодельная крипта = незаметная катастрофическая дыра.
- **Не переписывай WireGuard для прода.** Для продукта это нулевая пользовательская ценность и лишний риск. Свой транспорт — только учебный трек (A).
- **Не путай два трека.** Решай заранее, что ты сейчас делаешь — учишься или выпускаешь.
- **Не размазывайся.** Доведи фазу до критерия готовности прежде, чем браться за следующую.
- **Не переоценивай «ISP не читает трафик».** Контент уже закрыт HTTPS; твоя реальная добавка — метаданные и унификация. Будь точен в формулировках, это и есть зрелость.

## 8. Ресурсы для изучения

- **WireGuard whitepaper** — короткий, читается за вечер, это твой эталон дизайна.
- **Исходники WireGuard** (~4000 строк) — изучи *до* того, как писать своё.
- **`boringtun`** (Cloudflare) — userspace WireGuard на Rust, референс для Трека B.
- **Noise Protocol Framework spec** — официальная спецификация паттернов (IK и др.).
- **Tailscale blog «How NAT traversal works»** — лучшее объяснение hole punching и relay-фоллбэка.
- Документация крейтов: `snow`, `tun`, `tokio`, `x25519-dalek`.

## 9. Определение успеха

Минимальный успех — **Фаза 0**: пакет проходит сквозь тоннель, написанный твоими руками. Это уже доказывает, что ты понимаешь стек снизу доверху.

Продуктовый успех — **Фаза 5–6**: незнакомый человек разворачивает свою приватную mesh-сеть по твоей документации. С этого момента у тебя есть независимый законченный продукт, публичный артефакт компетенции и фундамент, на который можно навешивать что угодно дальше.

> Принцип всего проекта: не строй двигатель, чтобы потом искать под него машину. Машину — mesh-VPN — ты уже выбрал. Транспорт пишешь под неё, и каждый кусок кода имеет смысл.
